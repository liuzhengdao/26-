智能手表系统以STM32F103C8T6为主控核心，在其上运行FreeRTOS实时操作系统，通过I2C总线挂载OLED显示屏与MPU6050陀螺仪，通过USART串口连接HC-06蓝牙模块，通过GPIO外部中断接入EC11旋转编码器。电源部分由3.7V锂电池经AMS1117-3.3V稳压后为整机供电，TP4056模块负责锂电池的Type-C充电管理。

系统在软件层面划分为五个FreeRTOS任务：UI显示任务负责OLED界面刷新与多级菜单渲染，时间管理任务基于SysTick实现软件计时，传感器采集任务定期读取MPU6050数据并进行姿态解算，计步任务基于加速度峰值检测算法统计步数，蓝牙通信任务处理手机端的数据同步请求。各任务通过消息队列进行数据传递，通过二值信号量保护共享资源。

系统功能上实现了四大模块：时间显示模块在OLED主页面实时呈现当前时间；传感器模块展示三轴加速度、陀螺仪角速度及俯仰/横滚姿态角；计步模块通过MPU6050加速度数据实时计步并记录运动数据；蓝牙通信模块与手机上位机建立双向通信，支持时间同步、步数查询与传感器数据回传。旋转编码器作为人机交互输入设备，实现多级菜单的切换与功能选择。

    │      ├── 计步数据页面
    │      └── 设置页面
    │
    ├──► 时间管理任务 (优先级 4)
    │      │
    │      └── SysTick每秒更新RTC软件时钟
    │
    ├──► 传感器采集任务 (优先级 2)
    │      │
    │      ├── 读取MPU6050加速度/陀螺仪数据

    │

    │      │
    │      ├── 接收加速度数据
    │      ├── 峰值检测算法
    │      └── 更新步数计数器
    │
    ├──► 蓝牙通信任务 (优先级 2)
    │      │
    │      ├── 接收上位机命令
    │      ├── 发送时间/步数/传感器数据
    │      └── 命令解析与响应
    │
    └──► 编码器中断服务 (EXTI)
           │
           └── 解析旋转方向/按键，通过队列通知UI任务

**系统启动流程：**

系统上电后，首先执行HAL库初始化，完成系统时钟配置（HSE+PLL，主频72MHz）。随后初始化I2C1、USART1、GPIO中断等外设，依次调用SSD1306初始化序列和MPU6050初始化函数，完成外设自检。外设初始化通过后，创建FreeRTOS所需的消息队列和信号量等内核对象，然后依次创建五个功能任务。任务创建完成后，启动FreeRTOS调度器，由内核接管CPU控制权，各任务按优先级与时间片并发执行。

**各任务执行流程：**

UI显示任务（优先级3）以50ms为周期运行。首先检查编码器消息队列是否有新的操作事件：若收到旋转事件则移动菜单光标位置，若收到按键事件则进入当前高亮的子页面或确认选项。然后根据当前页面状态渲染对应界面：主页面显示时间（HH:MM:SS格式）和日期；传感器页面显示加速度、角速度与姿态角数值；计步页面显示今日步数和估算距离；菜单页面显示可选项列表并高亮当前光标项。渲染完成后通过I2C将显存数据写入OLED。

时间管理任务（优先级4，最高）以100ms为周期执行。利用SysTick滴答定时器产生的1ms基准时钟，在任务内累加毫秒计数器，当达到1000ms时进位秒，秒满60进位分，分满60进位时（24小时制）。支持通过蓝牙接收时间同步命令进行校时。

传感器采集任务（优先级2）以20ms为周期运行。每次通过I2C从MPU6050的ACCEL_XOUT_H寄存器（0x3B）开始连续读取14个字节，解析出三轴加速度、温度、三轴陀螺仪角速度的原始值并换算为物理量。基于加速度计数据通过反正切计算俯仰角和横滚角，再通过互补滤波算法融合陀螺仪积分值，得到平滑稳定的姿态角。加速度原始数据同时通过消息队列发送给计步任务。

计步任务（优先级1，最低）以20ms为周期运行。从消息队列接收传感器任务发来的三轴加速度数据，计算加速度合成幅值，维护一个32点的滑动窗口来动态追踪波形的峰值和谷值。动态阈值取峰值与谷值的均值。当加速度幅值自下而上穿越"动态阈值+最小阈值（0.3g）"并再次回落到动态阈值以下时，判定完成一步。为防止抖动误判，设置200ms的最小步间隔约束。每检测到有效一步，步数计数器加一并记录时间戳，用于后续计算步频和估算卡路里消耗。

蓝牙通信任务（优先级2）以事件驱动方式运行。通过USART1接收手机上位机发来的命令帧，帧格式为：2字节帧头（0xAA55）+ 1字节命令码 + 1字节数据长度 + N字节数据 + 1字节异或校验。接收到完整帧后，根据命令码执行对应操作：0x01为时间同步命令，解析数据中的时分秒并发送给时间任务进行校时；0x02为查询步数命令，将当前步数值打包回复；0x03为查询传感器数据命令，将最近一次MPU6050数据打包回复。也会主动按固定周期（如每5秒）向手机推送步数更新。

编码器输入通过GPIO外部中断处理。PA0（CLK）配置为下降沿触发中断，在中断服务函数中读取PA1（DT）电平判断旋转方向：DT为低表示顺时针（正向），DT为高表示逆时针（反向），据此设置旋转方向标志。PA2（SW）也配置为下降沿触发中断，标志按键按下。中断服务函数尽可能简短，仅记录事件标志，实际的事件处理在UI任务中通过查询消息队列完成。

**I2C总线电路：**

STM32的PB6（I2C1_SCL）和PB7（I2C1_SDA）分别连接到OLED模块和MPU6050模块的SCL和SDA引脚，两模块的SCL并联在一起、SDA也并联在一起。

**USART串口电路：**

STM32的PA9（USART1_TX）连接到HC-06蓝牙模块的RXD引脚，PA10（USART1_RX）连接到HC-06的TXD引脚。注意TX与RX需要交叉连接（发送对接收）。HC-06模块的工作电压为3.3V，可直接由系统3.3V供电，无需电平转换。

**旋转编码器电路：**

EC11旋转编码器有三个功能引脚：CLK（A相）接PA0，DT（B相）接PA1，SW（按键）接PA2。CLK和SW分别配置了外部中断（下降沿触发）。

**电源电路：**

TP4056充电模块输入接Type-C母座的VBUS（5V）和GND，输出端BAT+和BAT-分别接3.7V锂电池的正负极。TP4056的OUT+和OUT-（或直接取锂电池端电压）连接到AMS1117-3.3V模块的输入端。
#### 3.1.3 关键元件参数计算

**I2C上拉电阻计算：**

I2C总线在Fast Mode（400kHz）下，上拉电阻的取值需满足上升时间约束。标准计算公式为 Rp_max = tr / (0.8473 × Cb)，其中tr为上升时间上限（Fast Mode为300ns），Cb为总线总电容（估算OLED + MPU6050 + 走线 ≈ 30pF）。计算得 Rp_max ≈ 300ns / (0.8473 × 30pF) ≈ 11.8kΩ。下限由驱动电流决定，Rp_min = (VCC - VOL_max) / IOL，取VCC=3.3V、VOL_max=0.4V、IOL=3mA，计算得 Rp_min = (3.3-0.4) / 0.003 ≈ 967Ω。因此上拉电阻的合理取值范围为1kΩ至11.8kΩ，选择常用值4.7kΩ在范围内，兼顾了信号质量和功耗。

**AMS1117-3.3V 输入输出电容选择：**

AMS1117为LDO线性稳压器，为使输出稳定并防止自激振荡，输入端需并联10μF钽电容，输出端需并联22μF钽电容。同时在输入和输出端各并一只100nF高频瓷片电容，用于滤除高频噪声。LDO的功耗为 Pd = (Vin - Vout) × Iout，其中Vin为锂电池电压（3.7V~4.2V），Vout=3.3V，最大负载电流Iout估算为85mA，因此最大功耗 Pd = (4.2 - 3.3) × 0.085 ≈ 0.077W，AMS1117的SOT-223封装在无散热片条件下可承受约0.8W，裕量充足。


#### 3.1.4 PCB布局规划

本项目当前阶段采用面包板进行原型搭建与调试，暂不涉及PCB设计。后续再进行优化，PCB布局规划要点如下：

- 电源走线优先：AMS1117靠近电池接口放置，电源线宽不小于0.5mm，大电流回路面积最小化
- I2C走线：SCL和SDA走线长度尽量相等，远离高频信号源，过孔数量一致以减少信号偏差
- 模拟与数字分离：MPU6050为模拟敏感器件，其电源引脚就近放置100nF去耦电容
- 天线净空：HC-06蓝牙模块的板载天线区域下方不走铜、不铺地，保证无线信号质量
- 整体布局：采用双层板设计，顶层走信号线，底层完整铺地作为参考平面；STM32放在中心位置，各外设环绕布局以缩短走线

#### 3.2.2 关键算法描述

**算法一：软件实时时钟（Soft RTC）**

系统利用STM32的SysTick定时器（1ms中断一次）作为时钟基准源，在SysTick中断服务函数中累加毫秒计数器。毫秒计数器达到1000时归零并触发秒进位，秒计数器达到60时归零并触发分进位，分计数器达到60时归零并触发时进位，时计数器达到24时归零完成一天循环。年、月、日的日历管理根据各月天数规则进行进位处理，支持闰年判断。

该方案无需外接RTC芯片，节省硬件成本，精度取决于STM32外部8MHz晶振的精度（典型值±20ppm，每天误差约±1.7秒），对于智能手表应用可接受。通过蓝牙时间同步命令可进行周期性校准。

**算法二：峰值检测计步算法**

计步算法的核心思想是利用人体行走时产生的周期性加速度变化来统计步数。算法流程如下：

第一步，加速度幅值合成。将MPU6050输出的三轴加速度分量(ax, ay, az)按欧几里得范数合成为一维幅度信号：magnitude = sqrt(ax² + ay² + az²)。合成后的信号消除了传感器佩戴方向的影响，使任意方向的运动都能被检测。

第二步，滑动窗口与动态阈值。维护一个32个采样点的循环缓冲区，每收到一个新采样值就替换最旧的值。在每个窗口内跟踪幅值的最大值max和最小值min，动态阈值取二者均值：threshold = (max + min) / 2。动态阈值的优势在于能自适应不同走路速度和不同佩戴方式带来的加速度幅值变化，避免了固定阈值在不同场景下过灵敏或过迟钝的问题。

第三步，峰值检测与计步判定。维护一个"是否已跨过上升阈值"的标志位。当幅度从下方向上穿越"动态阈值 + 最小阈值偏移量（0.3g）"时，设置标志位为真；当幅度随后向下跌破动态阈值时，如果标志位为真，则判定完成一个完整步态周期，步数加一。随后重置窗口的max和min为当前幅度值，为下一轮检测做准备。

第四步，防抖处理。设置最小步间隔为200ms（相当于最高步频300步/分钟），在200ms内即使再次检测到有效步态也不计数，有效过滤了传感器抖动和异常尖峰造成的误判。

**算法三：互补滤波姿态解算**

MPU6050可同时输出加速度计数据和陀螺仪角速度数据。加速度计在静态时能准确推算俯仰角和横滚角（利用重力方向），但动态时受运动加速度干扰严重、噪声大；陀螺仪动态响应快、短时精度高，但长期积分会因零偏漂移而发散。互补滤波融合两者的优势：

加速度计推算的角度 = atan2(ay, sqrt(ax² + az²))（俯仰角）或 atan2(-ax, az)（横滚角）。

互补滤波公式：angle = α × (angle_prev + gyro_rate × dt) + (1-α) × accel_angle

其中α为陀螺仪权重系数，取0.98（经验值），dt为两次采样的时间间隔，gyro_rate为陀螺仪输出的角速度（需乘以灵敏度因子换算为°/s）。该算法计算量极小（仅需几次乘加运算），非常适合STM32F103C8T6这类资源受限的MCU，同时能获得平滑稳定的姿态角输出。

**任务列表：**

|----------|:------:|:--------:|----------|
| Task_Time | 4(最高) |100ms | 软件时钟走时；响应蓝牙时间同步 |

**优先级设计说明：**

时间任务设为最高优先级（4），确保时钟走时不会被任何其他任务延迟，保证时间显示的准确性。UI任务设为高优先级（3），因为界面刷新直接影响用户体验，50ms周期能保证20fps的流畅度。传感器任务和蓝牙任务同为正常优先级（2），传感器负责源源不断地产出数据供下游消费，蓝牙负责与外部设备通信。计步任务设为最低优先级（1），因为计步是计算密集型但实时性要求最低的功能，利用CPU空闲时间处理即可。

### 4.2 计步算法当前问题分析

目前计步算法的核心问题集中在以下方面：

1. 陀螺仪积分漂移导致仰角发散
当前姿态解算中，仰角通过陀螺仪角速度积分得到。由于陀螺仪存在零偏误差，即使传感器静止不动，每次积分也会累积微小误差，导致仰角随时间持续增大。当模拟走路摇晃时，运动加速度叠加在重力分量上，加速度计推算的参考角度失真严重，无法有效修正陀螺仪积分值，从而仰角变得非常大且无法恢复。

2. 计步算法的动态阈值适应性不足
在上述陀螺仪问题解决之前，加速度幅值存在叠加误差，导致计步算法中滑动窗口追踪的峰值谷值不准确，动态阈值偏移，步数统计出现漏计或过计。
